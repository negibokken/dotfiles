# タスクガイド

## 要件定義ガイド

要件定義書を作りたいと言った場合は、下記のフォーマットに従って作成するのを手伝ってください。

```markdown
## 概要

機能要件の概要を記載。

## 要件

1. 要件 1: 要件の名前
    - ユーザーストーリー「ユーザーは〜したい、なぜなら〜だからだ」の形式で記載。
    - 受け入れ基準: ユーザーストーリーが満たされていることを確認するための基準を記載。
2. 要件 2: 要件の名前
    - ユーザーストーリー「ユーザーは〜したい、なぜなら〜だからだ」の形式で記載。
    - 受け入れ基準: ユーザーストーリーが満たされていることを確認するための基準を記載。
```

それぞれ不明点を明らかにするように質問を行い、必要な情報を収集してください。
作成した要件定義書は`./claude/requirements/${要件のタイトル}.md`に出力してください。

## 開発ガイド

### 作業をはじめるときに

作業を始める前に `.claude/KNOWLEDGE.md` が存在する場合は必ず読み込み、過去の知見を把握してからタスクに取り組んでください。

作業をはじめるときに develop, main, master といった主要ブランチにいる場合は、最新のコミットをフェッチしてから修正のためのブランチを作ってから作業を進めてください。
ブランチの作成・切り替えは必ず `git wt` コマンドを使ってください（`git checkout` や `git switch` は使わないでください）。

```bash
# 最新をフェッチ
git fetch origin

# 新しいブランチのworktreeを作成して切り替え
git wt <branch-name>

# .envなどgitignoreされたファイルもコピーしたい場合
git wt --copyignored <branch-name>
```

なるべく細かい単位で commit してください。commit は指示がなくても適切なタイミングで行ってください。
作業後に develop や、main、master ブランチにいない場合は git push してください。

### コミットをするとき

コミットはできる限り細かい粒度でコミットしてください。
また、コミットする前に変更したファイルに対して format コマンドを実行してください。
基本的には 1 ファイルずつが原則ですが、意味がある場合は複数ファイルをまとめてコミットしても構いません。
また、コミットの順番もレビューをする人がわかりやすいようにコミットしてください。
最後にコミットの順番を見直し、分かりにくい場合は並び替えたり、コミットをまとめたりしてください。

### レビュー指摘に対する修正コミット

レビュー指摘（適）への修正は、フィックスアップコミットとして対象のコミットに紐づける形で作成してください。

```bash
# 対象のコミットを確認
git log --oneline

# フィックスアップコミットを作成（<hash> は修正対象のコミットハッシュ）
git commit --fixup=<hash>
```

**重要なルール:**

- スカッシュ（`git rebase -i --autosquash`）は **絶対に自動で実行しない**
- 通常のレビューフローでは、フィックスアップコミットをそのまま push してレビューアに確認してもらう
- スカッシュは **明示的に指示された場合のみ** 実行する

### PR を作成するとき

gh コマンドを使って PR を作成してください。
ただし、PR をは必ず draft にしてください。
PR のタイトルは英語で記載してください。
PR の本文には各コミットでどのようなことをしたか簡単に記載してください。

### 代替コマンド

一部のコマンドは rust で実装された代替コマンドを使用しています。

-   find は fd コマンドを使ってください
-   grep は ripgrep (rg) コマンドを使ってください

### git コマンドの注意事項

-   git コマンドに `-C` オプションは使わないでください。カレントディレクトリで実行するか、必要であれば `cd` を使ってください。
-   ブランチの作成・切り替えは `git checkout` / `git switch` を使わず、常に `git wt` を使ってください。
-   作業が完了したworktreeは `git wt -d <branch>` で削除してください（未マージの場合は `-D`）。

### プランを作るとき

プランを作るときは、以下のルールに従ってください。

1. `.claude/` ディレクトリ以下にタスク名のディレクトリを作成する（例: `.claude/tasks/<task-name>/`）
2. そのディレクトリ内に `PLAN.md` を作成し、プランの詳細を書き出す
3. 同じディレクトリに `TODO.md` を作成し、実行が必要なタスクを一覧化する

**PLAN.md の構成例:**

```markdown
# プラン: <タスク名>

## 概要
タスクの概要を記載。

## 背景・目的
なぜこのタスクが必要か。

## アプローチ
実装方針や手順の説明。

## 懸念事項・リスク
注意すべき点があれば記載。
```

**TODO.md の構成例:**

```markdown
# TODO: <タスク名>

## タスク一覧

- [ ] タスク1
- [ ] タスク2
- [ ] タスク3
```

タスクの進行に合わせて TODO.md のチェックボックスを更新してください。

### タスク実行中に得た知見の保存

タスク実行中に得た次のタスクに活かせそうな知見は `.claude/KNOWLEDGE.md` に構造的に保存してください。

- 新しい知見を得たら随時追記する
- カテゴリ別に整理して保存する
- 既存の項目と重複する場合は更新する

**KNOWLEDGE.md の構成例:**

```markdown
# KNOWLEDGE

## <カテゴリ1>

### <トピック>
- 知見の内容
- 注意事項

## <カテゴリ2>
...
```

### Todo が一度完了したら

Todo が完了したら、最後に`/retrospective` コマンドを実行して作業の振り返りをするように提案してください。

## 日付について言われたとき

タイムゾーンは、UTC+9（日本標準時）で考えてください。

今日の日付などはシステム情報を参照せずにコマンドを実行して正しい日付を確認してください。次のようなコマンドです。

```bash
date
```

昨日の日付を確認する場合は、次のようにします。

```bash
date -d 'yesterday'
```

## コーディングガイドライン

-   コードはできる限りシンプルに記載し、関数内にすべての処理を記載するのではなく、小さい粒度で関数化してください
    -   目安としては 1 つの関数が 20 行を超えないようにしてください
-   また、関数もファイルを分けるなどして、できる限りファイルサイズを小さくしてください
-   プロジェクトにフォーマットツールがある場合は、**変更したファイルにのみ**必ずフォーマットを実行してください
-   プロジェクトにリントツールがある場合は、**変更したファイルにのみ**必ずリントを実行してください

## GitHub のサブ issue の作成について

GitHub でまとまった issue を作るときは、親 issue と sub-issue が存在していることが望ましい。

gh コマンドの拡張である
https://github.com/yahsan2/gh-sub-issue を使ってサブ issue を作れるようにしている。

サブ issue を作るときは下記のコマンドを実行する。

```sh
gh sub-issue create --parent 123 --title "Task" --repo owner/repo
```

`/review`は日本語でレビューを行ってください。
